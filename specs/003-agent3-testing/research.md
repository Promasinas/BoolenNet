# Research Document: Agent 3 — 自动化测试

**Feature**: Agent 3 — 自动化测试
**Date**: 2026-06-06

## 1. File Monitoring on Windows

### Decision
使用 Windows `FindFirstFile` / `FindNextFile` API 进行轮询式文件变更检测，配合文件修改时间戳 (`ftLastWriteTime`) 记录。

### Rationale
- 项目宪法要求纯 C，Windows API 是 C 可调用接口
- `ReadDirectoryChangesW` 虽然提供事件驱动能力，但复杂度高、需要异步 I/O，不适合初期实现
- 轮询周期 1 分钟（规格 FR-001），时间戳比对足够满足检测需求
- 对每个文件维护 `路径 + 时间戳` 的快照，每次轮询对比差异

### Alternatives Considered
- **ReadDirectoryChangesW**: 事件驱动，实时性好但需要 IOCP/异步架构，复杂度过高
- **文件内容哈希（MD5/SHA）**: 准确但性能开销大，对大型源文件不友好
- **inotify (Linux)**: 项目目标平台为 Windows，不可用

## 2. Dynamic Makefile Generation

### Decision
使用模板填充策略：Agent 3 内嵌一个 Makefile 模板字符串，根据模块信息（UID、源文件路径、测试文件路径）填充模板变量，输出到测试目录。

### Rationale
- 每个测试模块的 Makefile 结构固定（编译目标 `all`、清理目标 `clean`），模板天然适合
- 避免引入额外的模板引擎依赖
- Makefile 只需三部分：源文件列表、编译标志（CFLAGS）、链接目标

### Template Structure
```makefile
# Auto-generated by Agent 3 for UID: {UID}
CC = gcc
CFLAGS = -Wall -Wextra -std=c11
TARGET = test_{UID}.exe
SRCS = {TEST_SRCS} {MODULE_SRCS}
OBJS = $(SRCS:.c=.o)

.PHONY: all clean
all: $(TARGET)

$(TARGET): $(OBJS)
	$(CC) $(CFLAGS) -o $@ $^

%.o: %.c
	$(CC) $(CFLAGS) -c -o $@ $<

clean:
	rm -f $(OBJS) $(TARGET)
```

### Alternatives Considered
- **CMake/autotools**: 引入外部依赖，违反纯 C 约束
- **手写 Makefile 每次全量生成**: 不够灵活，模板更易维护

## 3. Parallel Process Management

### Decision
使用 Windows `CreateProcess` API 为每个测试模块创建独立进程，编译和运行阶段各自独立。通过 `WaitForMultipleObjects` 管理并发上限（默认上限 = CPU 核心数 - 1，最少为 1）。

### Rationale
- `CreateProcess` 提供完整的进程隔离（独立工作目录、独立环境变量）
- `WaitForMultipleObjects` 支持多进程等待，适合并发控制
- 每个模块在独立的 `./test/uid_NNN/` 目录中运行 Makefile，编译输出和产物自然隔离
- 可配置并发上限防止系统过载（规格 Edge Case 要求）

### Concurrency Flow
```
检测到 N 个模块需要测试
  ↓
创建 N 个 Pipeline 结构体（各含独立状态）
  ↓
以并发上限 M 为窗口，启动 M 个 CreateProcess(make)
  ↓
WaitForMultipleObjects 等待进程完成
  ↓
编译成功的模块 → CreateProcess 运行 test_uid_NNN.exe
  ↓
收集输出 → 生成文档 → Git 提交
  ↓
窗口滑动，启动下一批
```

### Alternatives Considered
- **system() 调用**: 阻塞式，无法并行；输出捕获困难
- **popen() / _popen()**: 单向管道，无法同时捕获 stdout 和 stderr
- **线程池 + system()**: 线程模型比进程模型更适合 CPU 密集型编译任务，但 Make 本身会管理编译并发，多进程隔离更清晰

## 4. Git Integration

### Decision
使用 `system()` 调用 Git 命令行，通过临时批处理文件重定向输出。

### Rationale
- Git 命令行在 Windows（Git Bash / MinGW）下可用
- 避免引入 libgit2 等 C 库依赖
- 操作简单（仅 `git add` + `git commit`），无需复杂 Git API
- 错误处理：检查 `system()` 返回值，失败时记录到 `./docs/interact/`

### Commit Strategy
- 每次测试完成提交一次，消息格式：`[Agent3] Test uid_NNN (module_name) - YYYY-MM-DD HH:MM`
- 提交前检查 `git status --porcelain` 是否有变更，无变更跳过
- Git 锁竞争处理：串行化所有 Git 操作（使用全局互斥锁 `git_mutex`）

### Alternatives Considered
- **libgit2**: 功能完整但引入额外 C 库依赖，增加编译复杂度
- **直接操作 .git 目录**: 极不安全，容易损坏仓库

## 5. UID-Based Directory Naming

### Decision
测试目录命名为 `./test/uid_NNN/`，其中 NNN 是从交互指令 `## Target UID` 字段提取的 Layer UID 数值（`NNN` 为零填充的三位数字，与全项目 UID 格式一致）。

### Rationale
- 规格明确要求 Layer UID 为主标识
- UID 格式统一（零填充），目录排序自然
- `uid_NNN` 前缀便于区分测试目录与其他可能出现的目录类型
- 模块名记录在测试目录内的 `module_info.txt` 中作为辅助元数据

### Directory Layout
```
./test/
├── uid_001/
│   ├── module_info.txt      # UID=001, Module=bool_router
│   ├── Makefile             # Auto-generated
│   ├── test_main.c          # Test entry point
│   └── test_bool_router.c   # Module-specific tests
├── uid_002/
│   └── ...
└── uid_NNN/
    └── ...
```

### Alternatives Considered
- **纯模块名命名** (`./test/bool_router/`): 可读性好但模块可能重命名
- **UID + 模块名组合** (`./test/001_bool_router/`): 长度较长，分拆和管理复杂

## 6. Timeout Handling

### Decision
编译和运行阶段各自独立计时。超时值 MUST 由 Agent 2 在 `## Test Params` 的 `timeout` 字段显式指定（单位：秒），无默认值。Agent 3 MUST 拒绝缺少 `timeout` 参数的测试指令。超时后调用 `TerminateProcess` 强制终止。

### Rationale
- 无默认超时——Agent 2 作为请求方最了解模块特性和预期运行时间，必须显式提供
- 编译和运行阶段使用同一超时值，简化指令 schema 和实现
- `TerminateProcess` 是 Windows 上最可靠的进程终止方式
- 超时事件需记录到测试文档并标记状态
- 对应规格 FR-011、FR-015

### Implementation
```c
// timeout_seconds 从 ## Test Params 的 timeout 字段解析，缺失则拒绝指令
DWORD timeout_ms = timeout_seconds * 1000;
DWORD result = WaitForSingleObject(process_handle, timeout_ms);
if (result == WAIT_TIMEOUT) {
    TerminateProcess(process_handle, 1);
    log_timeout_event(module_uid, phase);
}
```

### Alternatives Considered
- **Job Objects**: Windows Job Objects 可设置进程超时，但 API 更复杂
- **SIGALRM (Unix)**: Windows 不支持，不可用

## 7. Error Reporting & Inter-Agent Communication

### Decision
Agent 3 通过文件系统与 Agent 1 和 Agent 2 通信。所有通知文档使用 Markdown 格式，放置于 `./docs/interact/` 目录。

### Document Types

1. **测试请求** (Agent 2 → Agent 3): 文件名 `test_request_uid_NNN.md`
2. **完成通知** (Agent 3 → Agent 1/Agent 2): 文件名 `test_done_uid_NNN.md`
3. **错误报告** (Agent 3 → Agent 1): 文件名 `error_uid_NNN.md`

### Rationale
- 文件系统通信在无网络、单机 C 项目中是最简单可靠的跨 Agent 通信方式
- Markdown 格式人机均可读，便于调试
- 文件名前缀区分消息类型，目录自然索引

### Alternatives Considered
- **管道/Named Pipe**: 需要 Agent 同时在线，耦合度高
- **共享内存**: 并发控制复杂，C 中实现成本高

## 8. Agent 3 Architecture Decision

### Decision
Agent 3 实现为一个 C 命令行程序 `agent3.exe`，主循环为轮询-执行模式：每 60 秒扫描 `./src/` 和 `./docs/interact/`，检测到任务后启动测试流水线。

### Program Structure
```
agent3.exe
├── main.c              # 主循环、轮询调度
├── scanner.c/h         # 文件系统扫描模块
├── pipeline.c/h        # 测试流水线管理（创建目录→编译→运行→文档→提交）
├── makefile_gen.c/h    # Makefile 模板生成
├── process_mgr.c/h     # Windows 进程管理（CreateProcess/超时/并发控制）
├── git_mgr.c/h         # Git 操作封装
├── reporter.c/h        # 测试文档生成、交互通知文档生成
├── config.c/h          # 配置读取（轮询周期、并发上限、编译超时等）
└── utils.c/h           # 通用工具函数
```

### Rationale
- 单进程、多模块结构，每个 .c 文件职责清晰
- 主循环轮询模型简单可靠，符合规格 FR-001
- 并发通过子进程实现（CreateProcess），主进程负责调度和文档生成

## 9. Heartbeat Mechanism

### Decision
Agent 3 每 60 秒（与轮询周期同步）写入 ISO 8601 时间戳到 `./docs/interact/.agent3_heartbeat`。
其他 Agent 检查该文件 mtime：超过 3 分钟未更新即判定 Agent 3 异常。

### Rationale
- 最低成本的存活性信号，无需额外 IPC 机制
- 文件系统时间戳是跨平台通用机制
- 3 分钟容忍度覆盖偶尔的文件系统延迟

## 10. Deduplication Window

### Decision
基于模块 UID 的内存哈希表记录上次测试触发时间戳。60 秒内同一 UID 的重复变更合并为一次测试请求。

### Rationale
- 与轮询周期（60 秒）对齐，实现简单
- 避免开发者连续保存文件时触发多次重复编译
- 超出窗口的新变更触发新一轮独立测试

## 11. No-Retry Policy

### Decision
测试运行失败后不执行自动重试。失败结果直接记录到测试文档，由 Agent 2 修复代码后重新触发。

### Rationale
- C 模块的确定性测试，失败通常是代码缺陷而非环境波动
- 自动重试会掩盖真实 bug、延长反馈周期
- 保持测试流水线简单可预测
